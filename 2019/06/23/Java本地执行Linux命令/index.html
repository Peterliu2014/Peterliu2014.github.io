Java本地执行Linux命令

发表于 2019-06-23 22:03:47 | 分类于 Java | | 阅读数
字数统计: 1.6k | 阅读时长 ≈ 6

Java提供了在Linux环境下执行命令的接口，其实现原理为：在JVM中fork一个执行命令的子进程，JVM通过管道与子进程通信，获得命令的执行结果。首先了解下Linux下管道通信的原理。

PIPE管道通信


1)父进程调用pipe开辟管道，得到执行管道的两个文件描述符，分别是表示读的fd[0]和表示写的fd[1]，简单来说，就是对应管道的读端和写端。读端负责从管道中读取数据，写端向管道中写数据。
2）父进程调用fork创建子进程，子进程复制父进程的两个文件描述符。
3）父进程关闭读端，子进程关闭写端（右侧图），这样父进程向管道中写数据，子进程读取数据，完成进程间的通信。
管道具有以下特点：

  • 管道是半双工的，只能有一个读端和一个写端，如果两个进程要实现双向通信，则需要建立两个管道。
  • 管道其实是内存中的一个环形队列，大小由BUFF_SIZE定义。当队列中已满时，写端会阻塞。读端读出数据之后，写端才可以继续写入。

    管道读写规则

    读规则

    1）如果所有指向管道写端的文件描述符都关闭了，但仍然有进程的读端从管道读取数据，那么读端读取过管道中剩余的数据之后，再次读取的话，linux提供的read方法会返回0（相当于读到文件末尾），表示读取已经完成，读取程序返回。
    2）如果有写端的文件描述符没有关闭，而写端也没有写入数据，则读取过剩余数据之后程序会阻塞。

    写规则

    1）如果读端所有文件描述符都已关闭，则写端写入的时候会收到信号SIGPIPE，通常写入的进程会异常终止。
    2）如果读端的文件描述符没有关闭，并且没有在读数据，那么把管道写满之后，写端进程会阻塞。

    生命周期

    应该是管道相关的所有读端和写端的文件描述符都关闭之后，管道生命周期结束。

有几点需要注意：

  • 有些命令需要shell才能解释执行，Java本身不能解释执行。比如“|”管道符号就需要/bin/bash的解释才难执行，因此才有command(“/bin/bash”, “-c”, “ls”)这种使用方式。
  • 使用redirectErrorStream(true)重定向错误输出到标准输出，便于对命令结果的处理。

    阻塞问题

    管道中的环形队列大小是有限的,如果子进程的输出结果比较多，就会导致管道被写满，进程因写入而阻塞。而父进程（JVM）中的waitFor一直阻塞等待子进程退出，进而导致父进程阻塞，产生互相等待的死锁现象。

进程超时退出

readLine也是个阻塞方法，如果标准输出流中没有数据（例如命令只输入“cat”就会一直卡着等待用户输入文件名），就会阻塞等待，这种场景下主进程还是会阻塞。因此有必要引入子进程超时退出的机制。

需要注意的几点：

  • waitFor(10, TimeUnit.SECONDS)会不断检测进程是否执行完成，执行完成则直接退出，返回false。否则，休眠等待，最多等待一定时间（这里是10s），则退出，并返回false。
  • exitValue()方法也是获得进程的退出值，但不是阻塞方法。如果调用时，进程还未结，则抛出异常IllegalThreadStateException。否则，返回进程的退出值。
  • isAlive()检测进程是否已执行完成，destroyForcibly()用于强制销毁进程（kill -9）。
