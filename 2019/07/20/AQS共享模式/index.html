Revision 1.73 acquireShared

共享模式中允许多个线程「同时获取」同步状态，即某一时刻允许「多个线程从acquireShared返回」,看下acquireShared具体实现：

在独占模式中，只允许一个线程获得同步状态，在同步状态已经被占用时，其它线程调用tryAcquire一定会失败。
而在共享模式中，允许多个线程获得同步状态，如果当前线程tryAcquireShared成功，那么其它线程如果有机会，或许也能获取到资源。因此与独占模式中acquire的一个不同点就是，在共享模式tryAcquireShared中，获取到资源并设置了head之后，根据tryAcquireShared「返回值」决定是否可以「唤醒」后继节点。与仅在release中「唤醒」相比，这样可以加快同步队列中线程的唤醒速度，这是一个「优化」。
除了直接在tryAcquireShared中唤醒后继线程之后，共享模式中acquire的实现与独占模式的逻辑基本相同。

Revision 1.73 releaseShared

可以看到，共享模式释放资源的逻辑与独占模式完全相同。文章开头提到的bug就是因为线程并发释放资源导致的。假设有4个线程，t1-t4,其中t1和t2已经获取到资源，t3-t4阻塞在同步队列中。

  • 时刻1: t1调用releaseShared释放资源，发现waitStatus为-1，因此唤醒t3，并将head的waitStatus设置为0，此时head是t3的前驱节点。
  • 时刻2: t3被唤醒，通过tryAcquireShared获取到资源，返回值是0（propagate），此时head仍然是t3的前驱节点。
  • 时刻3: t2调用releaseShared释放资源，读取到waitStatus为0，不满足唤醒条件，直接退出。这里「丢失了一次唤醒」。此时head仍然是t3的前驱节点。
  • 时刻4: t3调用setHeadAndPropagate，设置了新的head节点为t3。由于不满足「 if (propagate > 0 && node.waitStatus != 0) 」条件，不会唤醒后继节点。

可以发现，线程t4最终没有被唤醒，一直处于阻塞的状态。这就是文章开头提到的bug发生的原因。有同学可能会说t3调用releaseShared的时候不就把t4唤醒了吗？其实，在共享模式中，acquire和release两个操作不一定是成对出现在一个线程中的，可以参考复现bug的例子。

引入PROPAGATE状态

既然bug的原因是并发调用releaseShared，「丢失了唤醒」导致线程夯住的。那么解决办法应该就是这么防止唤醒丢失。
首先在Revision 1.74版本中引入了新的等待状态：PROPAGATE。

我们看看PROPAGATE是在哪里设置的。relaseShared方法也做了修改，修改后的实现如下：

为了解决并发release时可能导致的「唤醒丢失」，在doReleaseShared中有一个设置head为PROPAGATE的分支。现在我们再来看看之前的场景下是否还存在「唤醒丢失」的问题。

  • 时刻1: t1调用releaseShared释放资源，发现waitStatus为-1，因此唤醒t3，并将head的waitStatus设置为0，此时head是t3的前驱节点。
  • 时刻2: t3被唤醒，通过tryAcquireShared获取到资源，返回值是0（propagate），此时head仍然是t3的前驱节点。
  • 时刻3: t2调用releaseShared释放资源，读取到waitStatus为0，使用CAS将其设置为PROPAGATE(-2)。此时head仍然是t3的前驱节点。
  • 时刻4: t3调用setHeadAndPropagate，设置了新的head节点为t3。满足「propagate > 0 || h == null || h.waitStatus < 0」条件，唤醒后继节点t4。

由此可见，引入了PROPAGATE状态并修改相应代码之后，解决了并发调用releaseShared导致的「唤醒丢失」问题。
