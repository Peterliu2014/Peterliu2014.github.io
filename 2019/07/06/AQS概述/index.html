AQS概述

发表于 2019-07-06 20:39:21 | 分类于 Java | | 阅读数
字数统计: 2.1k | 阅读时长 ≈ 7

背景

在JDK5之前，synchronized是仅有的同步手段，也称为内置锁。这种锁在JVM层面实现，使用时需要synchronized修饰方法，或者修饰代码块。JDK5之后，引入了用于构建锁或其它同步组件的基本框架–AQS，通过AQS(AbstractQueuedSynchronizer)实现的锁使用起来更加灵活，并且支持更多的功能，例如支持公平、非公平锁，获取锁时的中断机制和超时机制。以下做一个简单的对比：

类别 synchronized AQS
存在层次 JVM级别 JUC包
功能 可重入 不可中断 非公平 可重入 可中断 可超时 可公平、非公平
锁的申请 如果持有锁的线程阻塞，其它申请锁的线程会死等 提供了超时机制和中断机制，不会死等
锁的释放(如果持有锁的线程崩溃) JVM会释放锁资源，不会导致死锁 可能会导致死锁。一般需要在finally块中释放锁
性能 JVM底层对synchronized做了很多优化，少量并发可使用 高并发可使用

AQS设计目标

acquire和release

提供操作同步状态的acquire和release方法，其中acquire方法阻塞当前线程，直到或者除非同步状态允许其运行。release方法通过某种方式修改同步状态，唤醒一个或者多个被acquire阻塞的线程，允许其继续运行。

运行模式

为了使AQS框架应用更加广泛，ASQ提供了两种模式：独占模式和共享模式。
独占模式：某个时刻，只允许一个线程通过acquire方法，其它线程都会阻塞到acquire方法上。
共享模式：某个时刻，允许多个线程通过acquire方法。

增强功能

阻塞和非阻塞：除了阻塞的acquire方法以外，提供一个非阻塞获取资源的方法（tryAcquire），允许线程尝试性的获取资源。
中断：提供中断机制的acquire，阻塞的线程可以被中断，直接退出。
超时：超时是在中断的基础上添加的，一定时间内没有获取到资源，则退出。

同步状态

使用整型变量state表示同步状态，并且使用volatile关键字保证可见性。

这个整数可以表现任何状态。比如，Semaphore 用它来表现剩余的许可数，ReentrantLock用它来表现拥有它的线程已经请求了多少次锁；FutureTask用它来表现任务的状态(尚未开始、运行、完成和取消)。

线程的阻塞和唤醒

Object的wait和notify方法提供了阻塞和唤醒的功能，但是这种方式有一定的局限性：

  • 必须在synchronized方法或者代码块中调用，使用方式不够灵活。
  • 必须在synchronized中调用，隐含着必须先获取到锁才能阻塞/唤醒线程，将获取内置锁和操作线程耦合在一起。

因此，AQS使用LockSupport实现线程的阻塞和唤醒。

park/unpark

LockSuppport提供了构建锁或其他同步器的线程阻塞原语。park/unpark模型实现线程粒度的阻塞和唤醒功能。park/unpark模型真正解耦了线程之间的同步，线程之间不再需要一个Object或者其它变量来存储状态。
park方法可以响应中断，即中断线程也可以唤醒阻塞的线程，这是实现AQS中断机制的关键。

许可机制

LockSupport通过“许可”（permit）机制，使用park/unpark实现线程的阻塞和唤醒。park申请许可，unpark释放许可，默认情况下许可是被占用的。可以将许可机制理解为值为1的信号量。
另外，如果unpark先于park被调用，park不会被阻塞（可以通过信号量来理解）。而Object的notify如果先于wait调用，则wait会阻塞。

同步队列

AQS对队列的管理是基于CLH队列来实现,是一个CLH变种形式。

  • head和tail都是懒加载的，只有当产生竞争，线程需要阻塞时才会创建。head指向获得资源的节点，tail指向队列的尾节点。
  • CLH队列是自旋锁，获取不到资源时会空旋。而这里同步队列获取不到锁，线程会阻塞，避免空旋浪费资源。
  • 由于需要使用unpark(Thread t)唤醒后继线程，所以在CLH队列的基础上，还添加了next字段，指向其后继节点。CLH中后继节点不断在前驱节点上自旋，因此不需要next字段。

    AQS具体实现

    实现AQS最好的方式是通过内部类继承AQS，然后实现其模板方法。五个模板方法（UnsupportedOperationException）是：
方法 功能
protected boolean tryAcquire(int) 独占式获取锁，实现该方法需要查询当前状态并判断同步状态是否和预期值相同，然后使用 CAS 操作设置同步状态
protected boolean tryRelease(int) 独占式释放锁，实际也是修改同步变量
protected int tryAcquireShared(int) 共享式获取锁，返回大于等于 0 的值，表示获取锁成功，反之获取失败
protected boolean tryReleaseShared(int) 共享式释放锁
protected boolean isHeldExclusively() 判断调用该方法的线程是否持有互斥锁

独占式获取与释放锁、共享式获取与释放锁的模板方法：

方法 功能
void acquire(int) 独占式获取锁，如果当前线程成功获取锁，那么方法就返回，否则会将当前线程放入同步队列等待。该方法会调用重写的 tryAcquire(int arg) 方法判断是否可以获得锁
void acquireInterruptibly(int) 和 acquire(int) 相同，但是该方法响应中断，当线程在同步队列中等待时，如果线程被中断，会抛出中断异常并返回。
boolean tryAcquireNanos(int, long) 在 acquireInterruptibly(int) 基础上添加了超时控制，同时支持中断和超时，当在指定时间内没有获得锁时，会返回 false，获取到了返回 true
void acquireShared(int) 共享式获得锁，如果成功获得锁就返回，否则将当前线程放入同步队列等待，与独占式获取锁的不同是，同一时刻可以有多个线程获得共享锁，该方法调用 tryAcquireShared(int)
acquireSharedInterruptibly(int) 与 acquireShared(int) 相同，该方法响应中断
tryAcquireSharedNanos(int, long) 在 acquireSharedInterruptibly(int) 基础上添加了超时控制
boolean release(int) 独占式释放锁，该方法会在释放锁后，将同步队列中第一个等待节点唤醒
boolean releaseShared(int) 共享式释放锁

著名实现类

  • ReentrantLock 是可重入的互斥锁（独占模式），同步状态来保存锁（重复）持有的次数。ReentrantLock 有公平锁和非公平锁两种模式，对应的，ReentrantLock内部有两个AQS的子类。
  • Semaphore表示信号量(共享模式)，同步状态来保存信号量的当前计数。它里面定义的acquireShared方法会减少计数，或当计数为非正值时阻塞线程；tryRelease方法会增加计数，可能在计数为正值时还要解除线程的阻塞。
  • CountDownLatch类使用AQS同步状态来表示计数。当该计数为0时，所有的acquire操作（acquire操作是从aqs的角度说的，对应到CountDownLatch中就是await方法）才能通过。
