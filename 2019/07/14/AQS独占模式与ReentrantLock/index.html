AQS独占模式与ReentrantLock

字数统计: 3.1k | 阅读时长 ≈ 13

CLH队列变体

在AQS概述的介绍中，我们知道需要三个组件的协作来实现AQS，即

  • 同步状态的原子性管理
  • 线程的阻塞和唤醒
  • 队列的管理

其中，前两个分别通过volatile、CAS以及LockSupport实现，即有现成的轮子可以使用。而队列的管理是在CLH队列上改进而来。

为什么不直接使用CLH

CLH队列中，后继线程会在前驱节点上自旋，直到前驱线程释放资源。AQS的设计初衷是作为并发包中其它工具的基础类，在高并发的环境中，大量线程为竞争锁资源而自旋必定浪费CPU资源，因此需要将CLH队列做一定的改进。

Node数据结构

简单总结一下，Node的主要数据成员有4个：waitStatus、prev、next和thread。

成员变量 作用
waitStatus 表示节点的等待状态
prev 表示节点的前驱节点
next 表示后继节点
thread 节点对应的线程

waitStatus有5中可能取值：

取值 描述
CANCEL(1) 当前线程取消申请资源，可能是因为超时或者中断，也可能是因为tryAcquire时发生了异常，是一个最终状态，不会再转化为其它状态
SIGNAL(-1) 当前线程的后继线程已经阻塞或者即将阻塞，线程释放资源时需要唤醒后继线程。这个值一般是后继线程入队后，阻塞之前设置的
CONDITION(-2) 条件队列中使用
PROPAGATE(-3) 用于将唤醒后继线程传递下去，这个状态的引入是为了完善和增强共享锁的唤醒机制。在一个节点成为头节点之前，是不会跃迁为此状态的
0 节点创建之后的初始值，表示无状态

与CLH相比，Node新增了三个字段：next、waitStatus、thread。正是通过这三个字段的协作才使AQS是一个阻塞锁，而非自旋锁。

独占锁acquire实现

acquire作为获得锁(不具有中断和超时机制)的入口方法，是一个阻塞方法，独占模式中某一时刻只允许一个线程从acquire返回。

实际获得同步状态的tryAcquire方法留给自类实现。返回值为true表示获取资源成功，否则失败，需要进入同步队列。

addWaiter的作用就是将线程包装为Node节点，然后挂到队尾，从方法退出时，传入的node节点一定已经挂到了队尾。
这里有个细节：入队时先设置当前node前驱节点为旧tail，然后使用CAS设置当前node为新的tail节点，成功的话则设置旧tail节点的next指向当前node。这就可以保证只要新的tail节点设置成功的话，一定可以向前遍历到head节点。 在之后unparkSuccessor方法唤醒线程时，只所以从后向前遍历，就是因为这个细节。

enq方法可以确保节点一定会入队。另外我们可以看到，同步队列其实时延迟初始化的，如果在acquire方法调用tryAcquire成功，就不会有入队的逻辑，进而，head和tail就不会被创建。

使用addWaiter方法完成node入队之后，进入acquireQueued方法。

上面提到acquire是一个阻塞方法，阻塞其实就是在这个方法里实现的，只有获取到了锁或者发生了异常，这个方法才会退出。
为什么是p == head && tryAcquire(arg)？
当前驱节点是head是才可能获取到锁，之所以还要tryAcquire是因为可能获得锁的线程还没有调用release释放锁资源。
如果没有获取到锁资源，则进入shouldParkAfterFailedAcquire方法

  • 如果前驱节点的等待状态已经为SIGNAL，直接返回true，表示当前节点可以放心阻塞。前驱节点释放时会将其唤醒。
  • 如果等待状态大于0，即为取消状态，则以当前节点为起点向前遍历，链接到一个非取消状态的节点，返回false。
  • 如果以上两者均不是，使用CAS设置前驱等待状态为SIGNAL，告诉它释放资源时，将自己唤醒，最后返回fasle。第二轮循环中就会命中第一个条件，返回true。

前驱节点为SIGNAL的话，就会调用parkAndCheckInterrupt方法。

真正的阻塞在这个方法里，通过LockSupport来实现的。这里有个需要注意的地方就是：unpark可以唤醒park退出阻塞状态，中断也可以将其唤醒，这也是AQS支持中断和超时机制的基础。

看一下具体的唤醒逻辑

如果head的next字段执行的node不会null，则直接唤醒相应的线程。否则，以tail为起点，向前遍历找到距离head最近的非取消节点，然后将其唤醒。
为什么是倒叙遍历呢？参考addWaiter方法可以发现某个时刻head节点的next可能会null，但是此时后续节点其实已经存在了。如果从前向后遍历就可能会导致一直阻塞，未被唤醒。

ReentrantLock

ReentrantLock是AQS的一个具体实现，基于独占模式实现的可重入锁。同步状态值表示获取锁的次数。通过内部类Sync继承AQS，然后通过FairSync和NonfairSync实现了公平锁和非公平锁。
