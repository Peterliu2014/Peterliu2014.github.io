AtomicLong与CAS

发表于 2019-05-04 19:54:29 | 分类于 Java | | 阅读数
字数统计: 1.6k | 阅读时长 ≈ 6

背景

并发环境下为了保证对共享资源的正确访问，通常采用对临界区加锁的方式来实现(例如synchronized，ReentrantLock)，这种加锁的方式假设线程每次读取共享资源时，其已经被其它线程做了修改，因此需要对临界区加锁,这种锁也称为悲观锁。

但是，还存在一种更加乐观的假设，即线程更新共享数据时，数据没有被其它线程修改。对数据更新时不加锁，但是会检测数据冲突，如果存在冲突(数据被其它线程修改过)，则采取相应的策略(例如直接报错或者重试)。这种思想可以称为乐观锁。java.util.concurrent下的atomic包中定义的一些类就是基于这种假设实现的一些线程安全类。

悲观锁会锁住共享资源，而乐观锁则不然，通过比较-交换(CAS)实现共享资源的并发访问。
悲观锁会导致死锁以及线程调度的开销（阻塞-唤醒），乐观锁本身没有加锁，自然不会产生死锁。

是什么

选择AtomicLong来介绍原子类。AtomicLong是线程安全的，通过一种无锁的方式实现的线程安全，底层基于CAS实现。

一个例子

AtomicLong可以实现并发环境下的计数器,一个简陋的例子(LongAdder用作计数器的性能更好，https://www.cnkirito.moe/java-concurrent-counter/)。

通过直接调用incrementAndGet完成自增，进而实现计数器的功能。不必使用锁做线程间同步，其实atomic原子类是使用CPU的CAS原子指令实现的线程同步。

考虑这样一种场景，线程A运行至1⃣️处，读取了最新的value值之后，时间片用完，线程被挂起。然后另外一个线程B修改了value值，最后线程A被调度继续运行，进入2⃣️，这时最新的value值与之前读取的value不相等，修改value值失败，返回false，进入循环体，继续运行。通过CAS这种不加锁的方式实现了并发访问的安全性。

CAS

CAS算法的过程：CAS(V,E,N)，其中V表示最新的内存值，E表示期望值，N表示最新值。如果V和E相等，那么可以将V设置为最新值N，此次CAS操作成功。否则，表示已经有线程对值进行了修改，此次操作失败。
多个线程同时使用CAS操作一个变量时，只有一个线程会成功，其余均会失败，失败的线程不会被挂起，可以允许继续重试。基于这个原理，CAS通过无锁的方式也可以发现其它线程对当前线程的干扰，并进行恰当的处理。

CAS存在的问题

ABA

CAS只是检查对象的值有没有变化，例如值由A->B->A,CAS是感知不到中间的变化过程的，认为没有线程修改过对象的状态。ABA问题的解决可以通过添加版本号来解决，例如1A->1B->2A,1A和2A并不同。JDK的Atomic包里提供了一个类AtomicStampedReference来解决ABA问题。

自旋问题

如果比较-交换一直不成功，线程就会一直自旋，这会导致不小的CPU开销。
